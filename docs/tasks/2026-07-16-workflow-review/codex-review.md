# Agent Workflow Skills Review — Codex 판정 전문

## 판정 기준

이 문서는 리뷰 당시의 `.ai/tmp` 기반 설계를 기준으로 한 판정이다. 검토 도중 감지된 artifact 경로 변경은 의도된 후속 수정으로 이해하며, 아래 판정에 소급 반영하지 않는다.

## 전체 판정

`CHANGES_REQUIRED`

전체 판정은 타당하다. 특히 원격 역할 활성화 계약 부재와 REQUIRED 4건은 실제 문서 간 계약 공백이다. 다만 일부 부가 지적은 심각도 또는 제안된 해결 방식에 조정이 필요하다.

설계의 기본 뼈대는 견고하다. 역할별 Mission/Prohibited가 대칭적이고, 오케스트레이터·아키텍트·임플리먼터의 쓰기 권한이 분리되어 있다. 긴 산출물을 파일로 전달하는 규칙과 통신을 논리 연산으로 추상화한 방향도 적절하다. 주요 위험은 원격 에이전트 활성화, 실패 후 복구, 실행 주체와 통신 계약의 불완전성에 집중되어 있다.

## BLOCKERS

### 1. 원격 역할 활성화 계약 부재

- 판정: 타당하며 `BLOCKER` 심각도가 적절하다.
- 근거: `BOOTSTRAP.md`의 진입 조건은 각 에이전트가 받는 메시지가 `워크플로워`로 시작하는 경우다. 반면 orchestrator 스킬은 원격 역할을 “Activate”하라고만 하고, 수신 에이전트가 역할 모드로 진입하도록 만드는 표준 메시지 형식을 정의하지 않는다.
- 분석: `SEND(role, message)`라는 운반 수단은 있으므로 통신 메커니즘 자체가 전혀 없는 것은 아니다. 정확한 결함은 “원격 역할 활성화 계약이 없다”는 것이다. 현재 상태에서는 오케스트레이터가 매번 임의의 프롬프트를 작성해야 하며, 수신 에이전트가 BOOTSTRAP과 해당 역할 스킬을 읽는다는 보장이 없다.
- 필요한 결과: 프로토콜 또는 BOOTSTRAP에 전용 활성화 envelope를 정의해야 한다. 예를 들어 `워크플로워 ROLE_ACTIVATE` 트리거, `TASK_ID`, 전체 역할 배정 또는 전용 활성 역할 정보, 역할명, 스킬 경로, orchestrator 반환 endpoint, 기대 상태 `ROLE_READY`를 포함해야 한다.
- 보정: 역할명·스킬 경로·반환 endpoint만 전달하면 현재 BOOTSTRAP의 “정확히 5개 역할 배정 파싱” 요구와 충돌할 수 있다. 원격 활성화 메시지에 전체 배정을 포함하거나, 별도의 원격 활성화 진입 규칙을 명시해야 한다.

## REQUIRED_CHANGES

### 1. `BLOCKED` 탈출 전이 추가

- 판정: 타당하다.
- 근거: 상태 전이표에는 `TDD_ASSESSMENT`, `IMPLEMENTING`, `TECHNICAL_ACCEPTANCE`에서 `BLOCKED`로 들어가는 전이는 있지만 `BLOCKED`에서 나오는 전이가 없다.
- 실패 영향: `TDD_BLOCKED`를 아키텍트에게 라우팅해 결정이 내려져도 합법적으로 작업을 재개할 수 없다.
- 필요한 결과: `STATE.md`에 `blocked_from` 또는 동등한 재개 정보를 기록하고, 최소한 다음 전이를 명시해야 한다.
  - `BLOCKED -> TDD_ASSESSMENT`
  - `BLOCKED -> IMPLEMENTING`
  - `BLOCKED -> TECHNICAL_ACCEPTANCE`
  - 재계획이 필요할 때 `BLOCKED -> PLANNING`
- 보정: 단순히 `BLOCKED -> 직전 상태`라고만 적으면 상태 이력과 재개 조건이 정의되지 않는다. “사용자 결정 대기” 역시 상태로 사용할 경우 명시적인 상태 및 전이가 필요하다.

### 2. TDD ASSESS 실행 주체 지정

- 판정: 타당하다.
- 근거: orchestrator 스킬은 `agent-workflow-tdd`를 ASSESS 모드로 호출하라고 하지만 누가 그 기술 판정을 수행하는지 지정하지 않는다. 오케스트레이터가 직접 수행하면 Technical design 및 Review judgment 금지와 충돌한다.
- 필요한 결과: 계획 PASS 후 아키텍트를 다시 활성화하고, 아키텍트가 TDD 지원 스킬의 ASSESS 모드를 수행하도록 고정하는 것이 가장 자연스럽다. 아키텍트는 코드·계획·Verification Requirements를 근거로 테스트 seam과 대체 검증을 판단할 수 있다.
- 보정: 임플리먼터를 실행 주체로 두면 “TDD assessment 없이는 구현을 시작하지 않는다”는 implementer 입력 계약과 순환할 가능성이 있다.

### 3. 컨텍스트 RESET 연산과 재식별 규칙 추가

- 판정: 타당하다.
- 근거: BOOTSTRAP은 같은 에이전트가 계획 리뷰어와 구현 리뷰어를 맡을 때 새 세션, clear 또는 동등한 초기화를 요구한다. 그러나 protocol은 `IDENTIFY`, `SEND`, `READ`, `WAIT_FOR`만 허용하므로 리셋을 표현할 수 없다.
- 실패 영향: 계획 리뷰 컨텍스트가 구현 리뷰에 남아 독립 리뷰가 훼손될 수 있다.
- 필요한 결과: `RESET_CONTEXT(role)` 또는 동등한 논리 연산, 성공 확인 상태, 실패 시 사용자 수동 리셋 요청, 리셋 후 `IDENTIFY` 재실행과 endpoint 갱신을 정의해야 한다.
- 보정: tmux의 `send-keys`로 `/clear`를 보내는 것만으로 일반 해결책이 되지는 않는다. `/clear`는 tmux 기능이 아니라 Codex·Claude 등 각 CLI의 기능이므로 어댑터가 CLI별 동작을 확인하거나 새 세션 생성 fallback을 제공해야 한다.

### 4. `ROLE_READY` 수집 시점 명문화

- 판정: 타당하다.
- 근거: BOOTSTRAP은 역할 담당자가 `ROLE_READY`를 반환한다고 하면서 준비 완료 시 “필요한 활성 역할”의 READY만 확인한다고 한다. 같은 출력은 `ACTIVE: orchestrator only`와 나머지 역할의 `DEFERRED`도 선언한다. README는 다시 “각 역할의 ROLE_READY 확인”이라고 기술한다.
- 필요한 결과: 준비 단계에서는 전체 역할의 배정과 endpoint 매핑만 기록하고 orchestrator의 READY만 확인한다. 아키텍트·계획 리뷰어·임플리먼터·구현 리뷰어의 `ROLE_READY`는 각 역할 최초 활성화 시 수집하고, 해당 단계 진입의 게이트로 사용한다고 명시해야 한다.

## DUPLICATION

### 역할 및 정책 중복

- 판정: 드리프트 위험은 타당하지만, 고정 역할 목록의 단일 출처를 CONFIG로 두는 제안은 최선이라고 단정하기 어렵다.
- 이유: 역할 집합은 사용자별 설정이라기보다 고정된 프로토콜 스키마다. BOOTSTRAP 또는 protocol을 규범적 출처로 두고 README는 참조·설명만 하는 편이 자연스럽다.
- 정책은 변경 가능한 설정이므로 `CONFIG.md`를 단일 출처로 삼고 BOOTSTRAP과 `WORKFLOW_READY`는 실제 읽은 값을 반영해야 한다.

### 어댑터 자동 선택 순서 중복

- 판정: 타당하다.
- 필요한 결과: 선택 순서는 `CONFIG.md`만 규범적으로 정의하고 BOOTSTRAP은 CONFIG를 따르도록 참조해야 한다.

### Orca compatibility 상태 부분집합

- 판정: 타당한 문서 결함이다.
- 근거: Orca 템플릿은 protocol의 상태 집합 일부만 다시 나열해 `TDD_*`, `*_CHANGES`, `*_BLOCKED` 등이 빠질 수 있다.
- 필요한 결과: 상태를 부분 재열거하지 않고 protocol의 전체 상태 집합이 손실 없이 왕복되어야 한다고 참조해야 한다.

## MISSING_GUARDS

### WAIT_FOR timeout·재시도·pane 사망 처리

- 판정: 타당하며 운영상 REQUIRED에 가깝다.
- 분석: Python 제어기를 후속으로 미루는 선택은 가능하지만, 현재 Markdown 하네스만으로 실행할 때 무한 대기를 막는 최소 규칙은 필요하다.
- 필요한 결과: 제한된 READ 재시도 또는 시간 상한, endpoint 재식별, pane/session 사망 확인, 복구 실패 시 `WORKFLOW_BLOCKED`와 사용자 보고를 정의해야 한다.

### 판정 문자열과 발신자 검증

- 판정: 타당하다.
- 분석: 캡처 결과에 상태 문자열이 존재하는지만 확인하면 구현 보고서 본문이나 과거 로그에 포함된 `IMPLEMENTATION_REVIEW_PASS`를 새 판정으로 오인할 수 있다.
- 필요한 결과: 현재 `TASK_ID`, 최상위 `STATUS`, `FROM`, `TO`, 배정 endpoint를 함께 대조하고 최신 미처리 메시지만 상태 전이에 사용해야 한다. `FROM` 필드만 신뢰하는 것으로는 충분하지 않다.

### artifact의 TASK_ID namespace

- 판정: 타당하다.
- 필요한 결과: 리뷰 당시 `.ai/tmp/<task-id>/`처럼 태스크별 하위 디렉터리를 사용해 이전 태스크 산출물과 섞이지 않도록 해야 한다.

### uninstall의 `python3` 선행 확인

- 판정: 타당하다.
- 근거: 관리 디렉터리를 먼저 삭제한 후 Python으로 bootstrap 블록을 제거하므로 Python이 없는 환경에서는 부분 제거 상태로 중단된다.
- 필요한 결과: 어떤 삭제도 하기 전에 `python3` 가용성을 확인하거나 Python 의존성을 제거해야 한다.

## NOTES

### uninstall 후 빈 instruction 파일

- 판정: 사실이며 무해하지만 정리 품질 문제다.
- 고려사항: bootstrap 블록만 있던 `AGENTS.md` 또는 `CLAUDE.md`를 제거 후 삭제할 수 있지만, 설치 전부터 존재했던 빈 파일과 설치기가 생성한 파일을 구분하지 못할 수 있으므로 사용자 파일 보존 정책을 우선해야 한다.

### “완성된 Orca 어댑터” 판별 기준

- 판정: 타당하다.
- 필요한 결과: placeholder 존재 여부, 필수 필드 충족, 사전 검증 명령 등 자동 선택 가능한 완료 조건을 정의해야 한다.

### 기본 계획 경로 `docs/plans/`

- 판정: 결함이라기보다 제품 선택이다.
- 분석: 계획은 임시 handoff보다 수명이 길고 승인된 계약 역할을 하므로 `.ai/tmp`로 옮기는 것은 오히려 위험할 수 있다. 대상 저장소의 `docs/` 점유를 피하려면 `.ai/plans/` 같은 추적 가능한 영속 경로는 고려할 수 있다.

### 프롬프트 기반 게이트 강제

- 판정: README의 “제어기는 나중에” 방침과 일관된 수용 가능한 트레이드오프다. 다만 현재 검증 스크립트가 이 의미적 계약을 강제하지 않는다는 한계는 남는다.

## `.ai/tmp` 변경 관련 우려 판정

### 1. 검색 불가시성

- 판정: 타당하며 실제 운용에서 가장 빨리 나타날 가능성이 높다.
- 이유: `.ai`는 hidden 경로이고 내부 `.gitignore`의 `*`로 인해 기본 `rg` 및 에이전트 검색에서 누락될 수 있다. 정확한 경로를 직접 전달하면 읽을 수 있지만 경로를 잃은 뒤 검색으로 복구하기 어렵다.

### 2. 워크트리 간 비공유

- 판정: 조건부지만 중요하다.
- 이유: 동일 디렉터리를 공유하는 pane에서는 문제가 없지만 역할별 worktree에서는 untracked·ignored 파일이 상대 worktree에 나타나지 않는다. 공유 절대경로 접근도 sandbox와 어댑터 구성에 따라 불가능할 수 있다.
- 필요한 결과: worktree 분리 어댑터는 공유 절대경로, 파일 복사, 메시지 본문 전송 또는 별도 artifact transport를 명시해야 한다.

### 3. uninstall 시 소실

- 판정: 타당하지만 임시 디렉터리 정책과 일치하는 의도된 위험이다.
- 필요한 결과: 제거 전에 진행 중 artifact가 삭제된다는 경고를 명확히 하거나, 태스크 진행 중 제거를 금지해야 한다.

### 4. 무한 축적

- 판정: 타당하다.
- 필요한 결과: `<task-id>` namespace와 태스크 완료 후 해당 디렉터리를 정리할 수 있다는 lifecycle 규칙을 함께 정의해야 한다.

## 검증 스크립트에 대한 결론

`scripts/validate.sh`가 통과하더라도 이번 리뷰를 반박하는 증거는 아니다. 현재 검증은 역할 스킬과 템플릿 존재, 설치·재설치·제거 동작 같은 구조를 확인한다. 원격 역할 활성화, 상태 전이, TDD 실행 주체, 발신자 검증, 컨텍스트 리셋 같은 의미적 계약은 검증하지 않는다.

## 최종 결론

핵심 5건은 모두 실제 수정 대상이며 원격 역할 활성화 계약 부재를 BLOCKER로 분류한 것도 합리적이다. `BLOCKED` 복구, TDD ASSESS 소유자, RESET 및 재-IDENTIFY, `ROLE_READY` 수집 시점은 REQUIRED 변경으로 유지한다. 중복·누락 guard·운영 notes도 대부분 타당하되, 역할 목록의 CONFIG 단일 출처화와 `docs/plans` 경로 문제는 제안의 강도를 낮춰 적용해야 한다.
