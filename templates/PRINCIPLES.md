# 개발 원칙

오케스트레이터를 제외한 네 역할이 따른다. 아키텍트는 설계할 때, 계획 리뷰어는 계획을 검증할 때, 임플리먼터는 코드를 쓸 때, 구현 리뷰어는 코드를 검증할 때 적용한다.

오케스트레이터는 기술적 판단을 하지 않으므로 적용 대상이 아니다.

이 파일은 저장소마다 조정할 수 있다. 워크플로 업데이트가 덮어쓰지 않는다.

## 읽기 쉬움이 최우선

복잡성, 코드 압축, 과도한 추상화보다 인간의 읽기 쉬움을 우선한다.

주석으로 복잡성을 설명하기보다 코드 구조 자체가 의도를 드러내도록 한다.

## YAGNI

현재 요구사항에 필요하지 않은 기능, 계층, 추상화는 추가하지 않는다.

실제 사용처가 확인되기 전까지 추가하지 않는다. 두 번째 사용처가 나타난 뒤 공통화를 검토한다.

중복이 실제로 반복되기 전에는 공통화하지 않는다.

가장 단순하게 동작하는 구현을 우선한다.

## 여백과 구조

코드의 여백, 줄바꿈, 들여쓰기와 블록 구분은 의미가 빠르게 드러나도록 구성한다.

관련 코드는 가깝게 두고, 서로 다른 의미 단위는 빈 줄로 구분한다.

불필요한 빈 줄은 제거하되, 논리적 단위 사이의 여백은 유지한다.

## 한눈에 읽히는 단위

한 줄에 과도한 로직을 넣지 않는다.

긴 메서드와 깊은 중첩을 피한다.

메서드 체이닝, 조건식, 생성자 인자가 한눈에 읽히지 않으면 여러 줄로 나눈다.
