# 워크플로워

사용자가 모든 탭에 같은 로스터를 뿌려 시작을 알린다.

```
워크플로워
코덱스: 아키텍트
테라: 플랜리뷰어, 구현리뷰어
루나: 임플리먼터
```

## 자기 역할 찾기

```bash
herdr tab list    # 전체 탭의 label, tab_id, pane_count
herdr pane list   # 전체 pane의 pane_id, tab_id
```

둘 다 목록 전체를 준다. 인자로 거르는 문법은 없다. 조인은 직접 한다.

`$HERDR_TAB_ID`와 같은 tab_id의 label이 자기 탭 이름이다. 로스터에서 그 이름의 역할을 읽는다.
자기 주소는 `$HERDR_PANE_ID`다 — 조회하지 않는다.

## 흐름

```
아키텍트 설계  → 플랜리뷰어  → PASS: 구현으로  | 지적: 아키텍트가 고쳐서 다시 리뷰
아키텍트 지시  → 임플리먼터  → 구현완료
아키텍트 지시  → 구현리뷰어  → PASS: 완료      | 지적: 아키텍트가 임플리먼터에게 수정 요청
```

판정은 `PASS`와 `지적` 둘뿐이다.

## 보내기

로스터의 탭 이름을 pane_id로 조인해서 보낸다.

```bash
herdr pane run w5:p2 '설계 리뷰 요청. docs/tmp/x.md. 회신: w5:p3'
```

작은따옴표로 감싸고 주소는 값을 직접 적는다. 큰따옴표를 쓰면 지시문 안의 `$()`, 백틱, `$VAR`가
보내는 쪽 셸에서 확장된다.

## 금지 하네스

역할이 시키는 일은 적지 않는다. 리뷰어는 리뷰할 줄 알고 임플리먼터는 구현할 줄 안다. 아래는 하지 말아야 할 것만이다.

**전원**

1. 자기 pane에 보내지 마라. 겸직한 역할은 그냥 이어서 한다. 자기한테 보내면 교착이다 — 보낸 메시지는 이번 턴이 끝나야 처리되는데, 턴은 답을 기다린다.
2. pane_id를 저장하지 마라. 보낼 때마다 조회한다.
3. 다른 herdr 세션으로 보내지 마라.
4. 주소를 짐작하지 마라. 없으면 되물어라.
5. 대상이 유일하지 않으면 보내지 마라. 같은 세션에서 그 label의 탭이 정확히 1개, 그 탭의 pane이 정확히 1개일 때만 보낸다. label이 없거나 겹치거나 탭이 쪼개져 있으면 멈추고 사용자에게 묻는다.

**아키텍트**

1. 자기 주소 없이 보내지 마라.
2. 금지 겸직이 섞인 로스터로 시작하지 마라. 자기 줄에 `아키텍트+플랜리뷰어`가 있거나 어느 줄에 `임플리먼터+구현리뷰어`가 있으면 사용자에게 재배정을 요구하고 멈춘다.
3. 어느 단계의 보고인지 모르면 짐작하지 말고 보낸 탭에 되물어라.
4. 플랜리뷰어가 보지 않은 설계를 임플리먼터에게 넘기지 마라. 구현의 근거가 되는 산출물만 해당한다.

**플랜리뷰어 · 구현리뷰어**

1. 이번 요청의 결과를 아키텍트에게 보고하기 전에 다음 단계 역할을 시작하지 마라.
2. 아키텍트 외에 보고하지 마라.
3. `PASS`와 `지적` 말고 다른 판정을 만들지 마라.
4. 직접 고치지 마라. 수정은 아키텍트를 거쳐 임플리먼터가 한다.
5. 자기가 만든 것을 리뷰하지 마라 — 구현리뷰어는 임플리먼터를 겸할 수 없다.

**임플리먼터**

1. 이번 요청의 결과를 아키텍트에게 보고하기 전에 다음 단계 역할을 시작하지 마라.
2. 아키텍트 외에 보고하지 마라.
3. PASS되지 않은 설계를 구현하지 마라.

**메시지**

메시지가 길면 `docs/tmp/`에 쓰고 링크로 보낸다.
